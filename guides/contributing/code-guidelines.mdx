Frappe HR follows the coding conventions used across the Frappe ecosystem. All contributions must pass the pre-commit hooks and test suite before being merged.

## Python

### Linter: ruff

Frappe HR uses [ruff](https://docs.astral.sh/ruff/) for Python linting and formatting. The configuration is in `pyproject.toml`:

```toml
[tool.ruff]
line-length = 110
target-version = "py310"

[tool.ruff.lint]
select = ["F", "E", "W", "I", "UP", "B", "RUF"]
ignore = [
    "B017",   # assertRaises(Exception) - should be more specific
    "B018",   # useless expression, not assigned to anything
    "B023",   # function doesn't bind loop variable
    "B904",   # raise inside except without from
    "E101",   # indentation contains mixed spaces and tabs
    "E402",   # module level import not at top of file
    "E501",   # line too long
    "E741",   # ambiguous variable name
    "F401",   # "unused" imports
    "F403",   # can't detect undefined names from * import
    "F405",   # can't detect undefined names from * import
    "F722",   # syntax error in forward type annotation
    "W191",   # indentation contains tabs
    "RUF001", # string contains ambiguous unicode character
]
```

### Formatting

```toml
[tool.ruff.format]
quote-style = "double"
indent-style = "tab"
docstring-code-format = true
```

Key formatting rules:
- **Double quotes** for strings
- **Tabs** for indentation (not spaces)
- Docstring code examples are formatted by ruff

### Import order

Imports are organized into sections with a specific order defined in `pyproject.toml`:

```toml
[tool.ruff.lint.isort]
section-order = [
    "future",
    "standard-library",
    "third-party",
    "frappe",
    "erpnext",
    "hrms",
    "first-party",
    "local-folder",
]
```

This means `frappe` imports come before `erpnext` imports, which come before `hrms` imports.

**Example:**
```python
import frappe
from frappe import _
from frappe.utils import getdate, today

from erpnext.setup.doctype.employee.employee import get_holiday_list_for_employee

from hrms.hr.utils import validate_active_employee
```

### Frappe conventions

- Use `frappe.get_doc()`, `frappe.get_list()`, and `frappe.db.get_value()` for database operations instead of raw SQL where possible.
- Use `frappe._()` for all user-facing strings to support translation.
- Raise `frappe.ValidationError` (or its subclass) for user-facing validation errors, not generic Python exceptions.
- Use `@frappe.whitelist()` to expose methods as REST API endpoints.

## JavaScript and Vue

### Formatter: Prettier

JavaScript, TypeScript, Vue, CSS, and SCSS files are formatted with [Prettier](https://prettier.io) (v3.1.0 via pre-commit).

Prettier is applied automatically by the pre-commit hook. To run it manually:
```bash
npx prettier --write "hrms/**/*.{js,ts,vue,css,scss}"
```

### Excluded from Prettier

The following paths are excluded from Prettier formatting (they may contain Jinja templates or are compiled bundles):

- `frontend/**` — The Frappe UI Vue SPA (has its own tooling)
- `hrms/public/dist/**` — Compiled asset bundles
- `hrms/templates/includes/**` — Jinja HTML templates
- Specific legacy JS files: `employee_promotion.js`, `employee_transfer.js`

## Commit message format

Frappe HR uses [Conventional Commits](https://www.conventionalcommits.org/) enforced by [commitlint](https://commitlint.js.org) with the `conventional-changelog-conventionalcommits` preset.

**Format:**
```
<type>: <subject>
```

**Rules:**
- `type` must be lowercase
- `type` must not be empty
- `subject` must not be empty

**Allowed types:**

| Type | When to use |
|---|---|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation changes only |
| `style` | Formatting changes that do not affect logic |
| `refactor` | Code restructuring without a feature or bug fix |
| `perf` | Performance improvements |
| `test` | Adding or updating tests |
| `build` | Changes to build tooling or dependencies |
| `ci` | CI pipeline configuration changes |
| `chore` | Miscellaneous maintenance tasks |
| `revert` | Reverting a previous commit |
| `patch` | Small patches that don't fit other categories |

**Examples:**
```
feat: add overtime slip auto-creation during payroll processing
fix: incorrect leave balance after carry-forward expiry
docs: update payroll settings field descriptions
refactor: extract salary component calculation into utils
test: add test for half-day attendance payroll deduction
```

<Warning>
Commits that do not match the conventional commit format will be rejected by the commitlint pre-commit hook.
</Warning>

## Testing

### Test runner

Frappe HR uses pytest via the Frappe test runner:

```bash
# Run all Frappe HR tests
bench --site hrms.localhost run-tests --app hrms

# Run a specific test module
bench --site hrms.localhost run-tests --app hrms \
  --module hrms.payroll.doctype.salary_slip.test_salary_slip

# Run a specific test method
bench --site hrms.localhost run-tests --app hrms \
  --module hrms.payroll.doctype.salary_slip.test_salary_slip \
  --test test_salary_slip_with_holidays
```

### Writing tests

- Test files are named `test_<doctype>.py` and live in the same directory as the doctype.
- Use `frappe.get_doc()` to create test documents and clean up with `frappe.db.rollback()` or teardown fixtures.
- Before-test setup is defined in `hrms/tests/test_utils.py` via the `before_tests` hook.
- Tests should be deterministic — avoid dependencies on external services or system clock without mocking.

**Example test structure:**
```python
import frappe
from frappe.tests.utils import FrappeTestCase

from hrms.payroll.doctype.salary_slip.salary_slip import make_salary_slip


class TestSalarySlip(FrappeTestCase):
    def test_half_day_deduction(self):
        # Arrange
        employee = create_test_employee()
        # Act
        slip = make_salary_slip(employee)
        # Assert
        self.assertEqual(slip.payment_days, expected_days)
```

## Submitting a pull request

<Steps>
  <Step title="Ensure all tests pass">
    ```bash
    bench --site hrms.localhost run-tests --app hrms
    ```
  </Step>
  <Step title="Ensure pre-commit hooks pass">
    ```bash
    pre-commit run --all-files
    ```
  </Step>
  <Step title="Push your branch">
    ```bash
    git push origin feat/your-feature-name
    ```
  </Step>
  <Step title="Open a pull request against develop">
    Open a PR on GitHub targeting the `develop` branch. Include:
    - A clear description of the change
    - Steps to reproduce (for bug fixes)
    - Screenshots for UI changes
    - Test coverage for new functionality
  </Step>
</Steps>

<Note>
PRs are reviewed by Frappe team maintainers. Review the [Pull Request Requirements](https://github.com/frappe/erpnext/wiki/Contribution-Guidelines) before submitting.
</Note>
